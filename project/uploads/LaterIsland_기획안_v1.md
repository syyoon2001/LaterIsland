# LaterIsland (나중에의 섬) 기획안 v1

## 1. 개요

### 배경 / 문제의식
콘텐츠가 여러 플랫폼(유튜브, 뉴스, 블로그, 팟캐스트, 책, 웹툰, 드라마 등)에 흩어져 있고,
사람들은 "나중에 봐야지" 하고 저장해둔 것들을 대부분 잊어버린다.
읽고 싶었던 것, 보고 싶었던 것들이 여기저기 흩어진 채 방치되고 있다.

### 제품 형태
- 웹 기반 PWA (앱스토어 출시 비용 회피, 기기에 설치 가능한 형태)
- 어떤 형태의 콘텐츠든(링크가 있는 것 / 없는 것 모두) 저장 가능

## 2. 핵심 기능 (MVP)

### 2.1 콘텐츠 저장 — 두 가지 입력 경로

**A. 링크 기반**
- URL 붙여넣기
- 페이지에서 제목 스크래핑 성공 시 → 자동으로 제목 채움 (입력창에서 수정 가능)
- 스크래핑 실패 시 → 사용자에게 제목 입력창 노출

**B. 제목 기반 (링크 없음)**
- 책, 웹툰, 드라마처럼 특정 URL이 없어도 되는 콘텐츠는 제목만 입력해서 저장

### 2.2 제목 / 요약 규칙
- 제목: 항상 존재 (스크래핑 또는 수동 입력), 언제든 수정 가능
- 요약: 기본값은 빈 상태, 필수 아님. 사용자가 원할 때만 직접 작성

### 2.3 AI 자동생성 버튼
- 버튼 클릭 시 제목 / 요약 / 카테고리(형식) / 태그(주제)를 AI가 한 번에 생성
- 이미 사용자가 입력해둔 값이 있어도 **덮어쓰기** (되돌리기 없음 → UI에 확인 경고 문구 필요)

### 2.4 카테고리(형식) & 태그(주제) 시스템
콘텐츠를 두 가지 축으로 분류한다. 형식은 콘텐츠당 하나로 정해지는 성격이라 **카테고리(단일 선택)**, 주제는 여러 개가 동시에 해당될 수 있는 성격이라 **태그(다중 선택)**로 구분한다.

**카테고리 (형식) — 단일 선택**
- 책/웹툰/드라마/영상/글 등, 콘텐츠당 하나만 선택
- 드롭다운으로 선택, 귀찮으면 **'기타'** 선택 가능
- 드롭다운에 **'+' 버튼**으로 사용자가 새 카테고리 직접 추가 가능

**태그 (주제) — 다중 선택**
- 한 콘텐츠에 여러 개의 태그를 붙일 수 있음 (예: '재테크' + '자기계발')
- 드롭다운/입력창에서 기존 태그 선택 또는 **'+' 버튼**으로 새 태그 직접 추가 가능
- 태그가 딱히 없으면 비워둬도 무방 (필수 아님)

**공용 풀 & AI 연동 (카테고리·태그 공통)**
- 사용자가 추가한 항목 ↔ AI가 생성하는 항목은 **하나의 공용 풀**을 공유
  - 사용자가 만든 카테고리/태그를 AI도 다음 콘텐츠 분류 시 재사용
  - AI 자동생성 버튼을 누르면 AI가 새 카테고리/태그를 만들어낼 수도 있음
  - (구현 시 유의점: AI에게 분류를 시킬 때 기존 카테고리/태그 목록을 프롬프트에 함께 제공해서 유사 항목 중복 생성 방지 필요 — 예: '드라마'와 '드라마물'이 따로 생기는 문제)

**태그 모음 화면**
- 태그만 모아보는 별도 화면 제공 (카테고리처럼 삭제/정리할 필요 없이 계속 누적)
- 정렬 기준: **최근에 클릭(선택)한 순서**
- 화면 내 **검색** 기능 제공 (태그 개수가 많아져도 원하는 태그를 바로 찾을 수 있도록)

### 2.4-1 콘텐츠 추가 창 구성
콘텐츠를 새로 추가할 때 입력창은 아래 순서로 구성한다.
1. **제목** (필수)
2. **링크** (선택, 없으면 제목만으로 저장 — 2.1 B 참고)
3. **요약** (선택)
4. **카테고리** (형식, 단일 선택)
5. **태그** (주제, 다중 선택)
6. **AI 자동생성 버튼** (제목/요약/카테고리/태그 일괄 생성)

### 2.5 콘텐츠 열람 방식
- 앱 내에서 원문/영상을 직접 끌어와서 보여주지 않음 (in-app reader 없음)
- 리스트에서 콘텐츠 요약 아래에 원본 링크를 작게 클릭 가능한 형태로 배치
- 사용자가 그 링크를 클릭하면 해당 플랫폼(기사 페이지, 유튜브 등)으로 직접 이동해서 열람
- 링크 옆에 작은 **복사 아이콘**을 두어, 클릭 시 링크 텍스트만 클립보드로 복사 가능 (바로 이동하지 않고 다른 곳에 붙여넣고 싶을 때 사용)
- 기사 같은 텍스트 콘텐츠뿐 아니라 동영상 등 어떤 형식이든 동일하게 외부 링크 클릭/복사 방식으로 처리

### 2.6 카테고리 · 태그별 보기
- 특정 카테고리(형식)를 선택하면 해당 콘텐츠만 모아보기
- 특정 태그(주제)를 선택하면 해당 태그가 붙은 콘텐츠만 모아보기 (한 콘텐츠가 여러 태그 뷰에 동시에 노출될 수 있음)

### 2.7 AI 검색
- 콘텐츠의 정확한 제목이 기억나지 않고 내용/태그(주제)만 어렴풋이 기억날 때, AI 기반 검색으로 콘텐츠 탐색

### 2.8 완료 처리
- 이미 읽었거나 실행한 콘텐츠는 "완료" 폴더로 이동

## 3. 데이터 모델 (초안)

**Content**
| 필드 | 설명 |
|---|---|
| id | 고유 ID |
| source_type | link / manual |
| url | nullable (수동 입력이면 없음) |
| title | 필수 |
| summary | nullable, 선택 항목 |
| category_id | 카테고리(형식) 참조, 단일 |
| status | pending / done |
| created_at | 생성일 |

**Category (형식) — 콘텐츠 : 카테고리 = N : 1**
| 필드 | 설명 |
|---|---|
| id | 고유 ID |
| name | 카테고리명 (예: 책, 웹툰, 드라마, 영상, 글, 기타) |
| created_by | user / ai |
| user_id | 소유 사용자 |

**Tag (주제) — 콘텐츠 : 태그 = N : M**
| 필드 | 설명 |
|---|---|
| id | 고유 ID |
| name | 태그명 |
| created_by | user / ai |
| user_id | 소유 사용자 |
| last_used_at | 태그 모음 화면 정렬(최근 클릭 순)에 사용 |

**Content_Tag (조인 테이블)**
| 필드 | 설명 |
|---|---|
| content_id | Content 참조 |
| tag_id | Tag 참조 |

## 4. 사용 AI (MVP 확정)

- **Google Gemini API 무료 티어** (Flash / Flash-Lite 모델) 사용
  - 요약, 카테고리/태그 분류, 검색 등 가벼운 작업에 충분한 성능
  - 무료 티어 유의사항:
    - 상업적 서비스 배포 시 약관상 제약이 있을 수 있어, 실제 다수 사용자 대상 서비스로 갈 경우 유료 전환 검토 필요
    - 무료 티어 사용 시 입력/출력 데이터가 모델 개선에 활용될 수 있음 (민감한 콘텐츠 저장 앱이라는 점에서 추후 고려 필요)
    - 요청 속도(RPM)·일일 요청수(RPD) 제한이 있어 사용자 수가 늘어나면 큐잉/제한 처리 로직 필요

## 5. 향후 고려 사항 (지금은 보류 — 나중에 다시 검토)

### 5.1 게임화 요소 — "섬" 컨셉 (구체화 전, 브레인스토밍 수준)
- 저장된 콘텐츠들이 "섬"에 쌓이고, 섬 주민들이 그로 인해 고통받는다는 은유
- 사용자가 콘텐츠를 정리(완료 처리)함으로써 섬을 정화하는 게임적 경험으로 설계
- 완료하지 않은 콘텐츠가 쌓일수록 섬에 안개/쓰레기가 늘어나는 비주얼
- 콘텐츠를 완료 처리할 때마다 섬 일부가 정화되는 연출
- 카테고리별로 섬의 다른 구역과 연결 (예: '책' 구역, '영상' 구역)
- 오래 방치된 콘텐츠는 섬의 "오염도"를 더 높이는 식으로 시각적 압박 부여
- 주민 캐릭터가 쌓인 콘텐츠 양에 따라 반응(대사/표정)하는 형태
- → MVP는 이 컨셉 없이 단순 리스트/폴더 형태로 출시하고, 핵심 저장·검색 기능이 안정화된 뒤 별도로 설계 필요

### 5.2 AI 무료 옵션 상세 정리

**Gemini 무료 티어 상세**
- Flash-Lite 모델 기준 하루 1,000~1,500회 요청, 분당 약 15회 정도 무료 제공
- Gemini 2.5 Flash / Flash-Lite는 분당 토큰(TPM) 100만 수준으로 넉넉한 반면, Gemini 2.5 Pro는 무료 티어에서 하루 50회로 크게 제한됨
- 요약/카테고리화 같은 가벼운 작업엔 Flash-Lite 정도로 충분
- **주의 1**: 무료 티어는 명시적으로 상업적 사용을 제외함 → 프로토타입/개인 프로젝트 단계엔 괜찮지만, 실제로 앱을 출시해서 여러 사용자에게 서비스하면 약관상 유료 전환 필요
- **주의 2**: 무료 티어에서는 프롬프트와 응답이 모델 개선에 사용될 수 있음 → 사용자 콘텐츠가 구글 학습 데이터로 쓰일 수 있다는 의미이므로 감안 필요

**완전히 서버비용 0원으로 가는 방법 — 브라우저 로컬 AI**
- PWA 특성을 살려서 transformers.js, WebLLM 같은 라이브러리로 요약/분류를 사용자 기기 브라우저에서 직접 돌리는 방법
- API 비용 자체가 안 들지만, 성능은 클라우드 LLM보다 떨어지고 저사양 기기에서는 느림
- Chrome에 내장된 Gemini Nano(온디바이스 AI)도 대안이 될 수 있으나, 크롬 브라우저에서만 동작해서 범용성은 떨어짐

**Gemini 외 추후 검토 가능한 무료 API (사용자 증가 시 대안/부하분산용)**
- **Groq** — 속도가 매우 빠름(Llama 3.3 70B 기준 초당 수백 토큰), 무료 티어 요청 수 제한 있음
- **Cerebras** — 무료 티어로 하루 약 100만 토큰까지 사용 가능, 카드 등록 불필요
- **OpenRouter** — 여러 무료 모델(Llama, DeepSeek, Qwen 등)을 하나의 API 키로 통합 사용 가능, 모델 비교/전환에 유리
- **Hugging Face Inference API** — 다양한 오픈 모델 무료 접근
- **Cloudflare Workers AI** — 하루 무료 요청 할당량 제공, 엣지 환경에 적합
- **Mistral (La Plateforme)** — 무료 티어에서 월 약 10억 토큰 제공(단, 학습 데이터 활용에 동의해야 하는 옵션 있음)
- **Cohere** — 검색/분류 특화, 임베딩(Embed) + 재랭킹(Rerank) 모델 제공 → AI 검색 기능 고도화 시 참고

**실용적인 추천 조합**
- MVP 단계: Gemini Flash-Lite 무료 티어로 시작 (요약, 카테고리화, 검색 임베딩 다 처리 가능)
- 사용자가 늘어나면: 유료 전환하거나, "사용자가 자기 API 키를 입력하는 방식(BYOK)"으로 운영 비용을 사용자에게 분산시키는 것도 read-it-later류 앱에서 흔히 쓰는 방법
